## Recommended sequence

Two independent tracks; do them in parallel if you have the hands:

- **Frontend track**: 001 → 002 → 006 → 005. 001 is the prerequisite for the
  unit-test harness; 002 builds the cache/export safety net; 006 (Playwright)
  automates the core-flow smoke checklist; 005 then refactors `App.tsx` against
  both nets. 006 is independent of 001/002 and can run in parallel with them — it
  uses its own Playwright runner, not Vitest.
- **Backend track**: 003 (standalone).
- **Cleanup**: 004 any time. If 003 is also being done, do 003 first so 004's
  Part B correctly leaves `session_secret_key` wired (004 has a guard for this).

If forced to pick a single highest-leverage item: **001** — it unblocks the
frontend's unit-test story and is low-risk. The single best guard against the
demo-style breakage (logout failing) is **006**.

## Dependency notes

- **002 requires 001** because it needs the Vitest harness that 001 installs.
- **005 requires 001 and 002** because decomposing the orchestrator component
  without the cache/export regression net (002) is too risky. **006 is the
  stronger net for 005**: its Playwright specs automate 005's manual smoke
  checklist (boot, logout, navigation, grader queue) and assert behavior via
  stable seams (`data-screen-label`, button titles) that the refactor does not
  change — so do 006 before 005 if you can.
- **006 depends on nothing**: Playwright drives the browser and asserts on
  behavior, not on `App.tsx`'s structure, so it is unaffected by 005 and is best
  written first as the safety net for it.
- **006 vs 007 — two layers, different jobs.** 006 is the fast, deterministic,
  every-PR CI net in **mock mode**: it catches frontend-wiring and routing
  regressions (button does nothing, view fails to transition, white screen).
  007 is a **local-only, manual** smoke against a **real Google session** and a
  prod-like single-origin build: it catches what mock cannot — that logout truly
  ends the session (the post-reload-stays-logged-out assertion). 007 is
  read-only + logout, never in CI, never stores a password (Playwright
  `storageState` from a one-time manual login). Run it before a demo/deploy.
- **004 Part B interacts with 003**: 003 gives `session_secret_key` a purpose
  (credential encryption). 004 Part B removes it *only if* 003 has not landed —
  the plan greps to decide. Do 003 first if both are planned.

## Findings considered and rejected

- **PostingPiP empty-queue crash** (`queue[Math.min(idx, -1)]` →
  `undefined.final_score` in `PostingPiP.tsx:98-100`): not reachable. The call
  site guards it (`pipQueueRef.current.length > 0` at `GraderWrap.tsx:331`) and
  the "Postagem guiada" button is disabled when `pipQueue.length === 0`
  (`GraderWrap.tsx:250`). Defensive-only; not worth a plan.
- **id_token decoded without signature verification**
  (`routers/auth.py:184-189`): by-design. The token comes directly from Google's
  token endpoint over TLS in the server-to-server `fetch_token` exchange, not
  from the client — verifying its signature locally adds nothing. Standard
  practice.
- **Secrets hygiene**: clean. `.gitignore` covers `.env`, `.env.*`, `.tokens/`,
  `*.db`, `.cache/`; `git ls-files` shows no tracked secrets. No action.
- **Static catch-all / submission-preview path traversal**: already defended
  with `Path.resolve()` + `is_relative_to`, a MIME allowlist, and `nosniff`
  (`main.py:189-203`, `routers/grading.py:1070-1136`). No action.
- **Backend authorization (IDOR)**: every grading endpoint scopes by
  `user_email` via `_get_owned_job`, including inside the SSE worker threads
  (`routers/grading.py`). No action.
